Frequently Asked Questions

Keyboard accessibility means every feature and interaction on a page can be reached and operated using only a keyboard — no mouse required. This benefits people who cannot use a pointer device, power keyboard users, and those who rely on switch access or voice control software.

The core requirements are: all interactive elements reachable via Tab, a clearly visible focus indicator, and no keyboard traps.

WCAG 2.2 Level AA is the internationally recognised standard for digital accessibility. At this level, you must satisfy all Level A and Level AA success criteria, including:

  • Text contrast of at least 4.5:1 (3:1 for large text)
  • Visible focus indicators that meet a 3:1 contrast threshold
  • All functionality operable by keyboard
  • No content that flashes more than three times per second
  • Drag movements have a pointer alternative (SC 2.5.7)
  • Target size of at least 24×24 CSS pixels (SC 2.5.8)

The native HTML <details>/<summary> element provides built-in disclosure behaviour and is a good default. A <button>-based pattern is preferred when you need:

  • Consistent cross-browser accessible name announcements
  • Full control over animation and transition timing
  • Custom styling that is reliable across all browsers
  • Integration with JavaScript-driven state management

Both approaches are valid; choose based on project constraints and existing component patterns.
